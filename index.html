Home / Introduction / Philosophy

The Philosophy of Auth Architecture

Before writing a single line of code, understand the thinking behind good authentication architecture.

1. Nothing Should Exist Without a Reason

Every file in your project exists because it solves a problem. If you cannot explain why a file exists, it probably should not exist. This is the foundation of clean architecture.

2. Architecture Evolves

You do not start with a perfect architecture. You start with a simple login button, and you let the architecture grow naturally. Each new file is a response to a new problem.

3. Separation of Concerns

One file should have one responsibility. The login screen should NOT contain API calls. The API service should NOT contain UI logic. The hook should NOT contain business logic.

LoginScreen
↓ uses
useAuth() Hook
↓ reads from
AuthProvider (Context)
↓ calls
AuthService
↓ uses
Firebase / Supabase SDK

4. Data Flows One Way

In a well-designed auth system, data flows in one direction:

1User ActionUser taps Login button
2ScreenCaptures input, calls hook
3HookManages local state, calls context
4Context/ProviderUpdates global auth state
5ServiceCalls external auth SDK
6SDK/APIAuthenticates, returns token
7StoragePersists token securely
8NavigationRedirects to Home screen

📝 Key Takeaways

  • Every file must have a clear reason to exist
  • Architecture starts simple and grows with needs
  • Separation of concerns is non-negotiable
  • Data should flow in one direction
  • The goal is maintainability, not cleverness

The Empty Project

There is no authentication. There are no files. The journey begins from nothing.

Every real project starts here: a blank Expo project. No authentication. No login screen. No services. Nothing.

The Project

my-app/ ├── app/ │ ├── _layout.tsx │ ├── index.tsx │ └── home.tsx ├── assets/ ├── app.json ├── package.json └── tsconfig.json

The Problem

A user opens the app and sees the home screen directly. There is no login. Anyone can access everything.

⚠️ Security Risk

Without authentication, there is no concept of "who is using the app." Every user sees the same content. Data cannot be personalized. Nothing is protected.

The Question

Where should login logic go?

The naive approach: put everything in index.tsx.

app/index.tsx TSX — BAD EXAMPLE 
import { useState } from 'react';
import { View, TextInput, Button, Alert } from 'react-native';
import { signInWithEmailAndPassword } from 'firebase/auth';

export default function LoginScreen() {
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');
  const [loading, setLoading] = useState(false);

  const handleLogin = async () => {
    setLoading(true);
    try {
      await signInWithEmailAndPassword(auth, email, password);
      // TODO: navigate somewhere
    } catch (error) {
      Alert.alert('Error', error.message);
    }
    setLoading(false);
  };

  return (
    <View>
      <TextInput value={email} onChangeText={setEmail} />
      <TextInput value={password} onChangeText={setPassword} />
      <Button title="Login" onPress={handleLogin} />
    </View>
  );
}

Why This Is Bad

  • Firebase SDK imported in a UI component — mixing concerns
  • Authentication state is local — other screens cannot know if user is logged in
  • No token storage — user must login every time they open the app
  • No navigation control — how do other screens know auth state?
  • Not reusable — cannot test, cannot swap Firebase for Supabase
💡 What We Need

Authentication logic separated from the UI. A service layer that handles the actual authentication. But we will not create that yet — first we need to feel the pain of NOT having it.

📝 Key Takeaways

  • Every project starts from nothing
  • The naive approach puts auth logic directly in screens
  • This creates problems: no shared state, no token persistence, no testability
  • These problems drive the need for the next stage

First Login Attempt

What Changes

We realize that putting Firebase logic inside the screen is wrong. So we create a service file — a file whose only job is to talk to Firebase.

The Service File

src/services/auth.service.ts TypeScript 
import { signInWithEmailAndPassword, createUserWithEmailAndPassword }
  from 'firebase/auth';
import { auth } from '../config/firebase';

export const AuthService = {
  async login(email: string, password: string) {
    const result = await signInWithEmailAndPassword(auth, email, password);
    return result.user;
  },

  async register(email: string, password: string) {
    const result = await createUserWithEmailAndPassword(auth, email, password);
    return result.user;
  },

  async logout() {
    await auth.signOut();
  }
};

What This File Does

AspectDetail
PurposeSingle place that talks to Firebase Auth SDK
Responsibilitieslogin, register, logout — nothing else
Inputsemail, password
OutputsUser object or throws error
Who imports itLogin screen, Register screen
Who calls itScreens that need auth actions

The Problem That Remains

The service works, but the screen still needs to know if the user is currently logged in. If the user refreshes the app, the screen has no idea. The login state lives and dies inside the component.

💡 Next Step

We need a way to share authentication state across the entire app. A single source of truth. That is what Context provides.

📝 Key Takeaways

  • The service file isolates Firebase/SDK logic from the UI
  • Screens call the service but do not know how it works internally
  • If we switch to Supabase later, only this file changes
  • Problem remaining: no shared auth state across the app

storage/auth.storage.ts

src/storage/auth.storage.ts TypeScript 
import * as SecureStore from 'expo-secure-store';

const TOKEN_KEY = 'auth_token';
const USER_KEY = 'auth_user';

export const AuthStorage = {
  async saveToken(token: string) {
    await SecureStore.setItemAsync(TOKEN_KEY, token);
  },

  async getToken(): Promise<string | null> {
    return SecureStore.getItemAsync(TOKEN_KEY);
  },

  async saveUser(user: User) {
    await SecureStore.setItemAsync(USER_KEY, JSON.stringify(user));
  },

  async getUser(): Promise<User | null> {
    const data = await SecureStore.getItemAsync(USER_KEY);
    return data ? JSON.parse(data) : null;
  },

  async clear() {
    await SecureStore.deleteItemAsync(TOKEN_KEY);
    await SecureStore.deleteItemAsync(USER_KEY);
  }
};
⚠️ Why Not AsyncStorage?

AsyncStorage stores data in plain text. Auth tokens are sensitive. expo-secure-store uses the device's secure enclave (iOS Keychain / Android Keystore) and is encrypted at rest. Always use SecureStore for authentication data.

📝 Key Takeaways

  • types/ ensures type safety across the entire auth system
  • config/ centralizes configuration (Firebase keys, API URLs)
  • storage/ handles secure persistence of tokens and user data
  • Services now depend on storage and types
  • Problem: screens still manage auth state locally

Hooks & Context

The moment auth state becomes global. Every screen can now know who is logged in.

Why Context?

Without context, every screen that needs to check if the user is logged in must pass props through every component. This is called prop drilling and it becomes unmanageable quickly.

Without Context (prop drilling):

App.tsx→Nav→Tab→Screen

With Context (direct access):

AuthProvider→Any Screen

New Files

src/ ├── types/ ├── config/ ├── storage/ ├── services/ ├── context/ ← NEW │ └── auth.context.tsx ← NEW ├── hooks/ ← NEW │ └── useAuth.ts ← NEW └── app/

context/auth.context.tsx

src/context/auth.context.tsx TSX 
import { createContext, useContext, useState, useEffect }
  from 'react';
import { User, AuthState } from '../types/auth.types';
import { AuthService } from '../services/auth.service';
import { AuthStorage } from '../storage/auth.storage';

const AuthContext = createContext<AuthState>(null!);

export function AuthProvider({ children }: { children: React.ReactNode }) {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // Check for existing session on mount
    const checkAuth = async () => {
      const storedUser = await AuthStorage.getUser();
      if (storedUser) setUser(storedUser);
      setLoading(false);
    };
    checkAuth();
  }, []);

  const login = async (email: string, password: string) => {
    const u = await AuthService.login(email, password);
    await AuthStorage.saveUser(u);
    setUser(u);
  };

  const logout = async () => {
    await AuthService.logout();
    await AuthStorage.clear();
    setUser(null);
  };

  return (
    <AuthContext.Provider value={{ user, loading, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}

export const useAuth = () => useContext(AuthContext);

Why Context + Hook Pattern?

✅ Do
  • Create a useAuth() hook that wraps useContext
  • Export the hook for clean imports
  • Keep provider and hook in the same file or nearby
❌ Don't
  • Use useContext(AuthContext) directly everywhere
  • Export the raw context object
  • Forget to handle the "not inside provider" case

📝 Key Takeaways

  • Context eliminates prop drilling for auth state
  • useAuth() hook provides a clean API for screens
  • AuthProvider initializes by checking stored session
  • Login updates both storage and context state
  • Problem: the Provider needs to wrap the entire app — but navigation depends on auth state too

Providers & Navigation Guard

Navigation now depends on auth state. Protected routes appear.

The Problem

Navigation needs to know if the user is logged in to decide which screens to show. But navigation lives in app/_layout.tsx, which runs BEFORE any screen.

providers/ — Why a Separate Folder?

Context files define WHAT state exists. Provider files wrap the app and provide that state. Separating them keeps files focused and composable.

src/ ├── types/ ├── config/ ├── storage/ ├── services/ ├── context/ ├── hooks/ ├── providers/ ← NEW │ └── RootProvider.tsx ← NEW └── app/ └── _layout.tsx ← Updated

import { AuthProvider } from '../context/auth.context';

app/_layout.tsx

app/_layout.tsx TSX 
import { Stack } from 'expo-router';
import { RootProvider } from '../src/providers/RootProvider';
import { useAuth } from '../src/context/auth.context';
import { ActivityIndicator, View } from 'react-native';

function AuthLayout() {
  const { user, loading } = useAuth();

  if (loading) {
    return (
      <View style={{ flex: 1, justifyContent: 'center', alignItems: 'center' }}>
        <ActivityIndicator size="large" />
      </View>
    );
  }

  return (
    <Stack>
      {user ? (
        <Stack.Screen name="home" options={{ headerShown: false }} />
      ) : (
        <>
          <Stack.Screen name="index" options={{ headerShown: false }} />
          <Stack.Screen name="register" options={{ headerShown: false }} />
        </>
      )}
    </Stack>
  );
}

export default function RootLayout() {
  return (
    <RootProvider>
      <AuthLayout />
    </RootProvider>
  );
}
✅ Architecture Milestone

The navigation now respects auth state. Unauthenticated users see the login screen. Authenticated users see the home screen. This is a real auth guard.

📝 Key Takeaways

  • Providers wrap the app and provide context to all children
  • Navigation uses useAuth() to decide which screens to render
  • Loading state prevents flickering during session check
  • The architecture now has a complete auth guard
  • Problem: no way to use repositories for complex data operations

File-by-File Summary

types/auth.types.ts

Purpose: Define all TypeScript interfaces for authentication.

Who uses it: Every other auth file.

Without it: No type safety, easy to pass wrong data.

config/firebase.ts

Purpose: Initialize and export Firebase app, auth, and Firestore instances.

Who uses it: auth.service.ts, user.service.ts

Without it: No connection to Firebase.

storage/auth.storage.ts

Purpose: Securely persist tokens and user data using expo-secure-store.

Who uses it: auth.context.tsx, auth.service.ts

Without it: User must login every app restart.

services/auth.service.ts

Purpose: Communicate with Firebase Auth SDK. login, register, logout, resetPassword.

Who uses it: auth.context.tsx

Without it: UI directly calls Firebase SDK (bad practice).

repositories/auth.repository.ts

Purpose: Abstract data access. Maps between Firebase responses and app types.

Who uses it: auth.service.ts (optional layer)

Without it: Service directly handles raw SDK responses.

context/auth.context.tsx

Purpose: Define and provide auth state to the entire component tree.

Who uses it: useAuth() hook, all screens

Without it: No shared auth state, prop drilling.

hooks/useAuth.ts

Purpose: Clean API to access auth context. Handles edge cases.

Who uses it: Every screen and component.

Without it: Direct useContext calls everywhere.

providers/RootProvider.tsx

Purpose: Compose all providers into one wrapper. Keeps _layout.tsx clean.

Who uses it: app/_layout.tsx

Without it: _layout.tsx nests multiple providers directly.

components/auth/LoginForm.tsx

Purpose: Reusable login form UI. Handles email/password input and validation.

Who uses it: Login screen

Without it: Login form code duplicated across screens.

app/index.tsx (Login Screen)

Purpose: Orchestrate login flow. Uses LoginForm, calls useAuth().

Who uses it: Navigation (shown when not authenticated)

Without it: No way for users to log in.

  • Every folder has a clear, single responsibility
  • Data flows from screens → hooks → context → services → SDK
  • Storage persists tokens across app restarts
  • Navigation guards protect routes based on auth state
  • The architecture is now production-ready

Complete Folder Structure

Every folder explained: purpose, when to create, who uses it, common mistakes.

📂 src/types/

Purpose: All TypeScript type definitions and interfaces.

When to create: FIRST. Before any other folder. Types define the contract for everything.

Who uses it: Every file in the project.

Example files: auth.types.ts, user.types.ts, navigation.types.ts

Common mistake: Defining types inside components instead of here.

Without it: No type safety. TypeScript becomes useless.

📂 src/config/

Purpose: External service initialization (Firebase, Supabase, etc.).

When to create: When you first add an external service.

Who uses it: Services layer (auth.service.ts, user.service.ts).

Example files: firebase.ts, supabase.ts, clerk.ts

Common mistake: Putting API keys directly in service files.

📂 src/constants/

Purpose: Application-wide constants that do not change.

When to create: When you have magic strings or numbers repeated across files.

Who uses it: Any file that references fixed values.

Example files: auth.constants.ts (token keys, error messages)

Common mistake: Hardcoding the same string in 5 different files.

📂 src/utils/

Purpose: Pure utility functions with no side effects.

When to create: When you need email validation, date formatting, etc.

Who uses it: Components, hooks, services.

Example files: validation.ts, format.ts

Common mistake: Putting business logic in utils (use services instead).

📂 src/storage/

Purpose: Secure local data persistence (tokens, user sessions).

When to create: When you need to persist login across app restarts.

Who uses it: Context, Services.

Example files: auth.storage.ts, settings.storage.ts

Common mistake: Using AsyncStorage for sensitive data. Always use expo-secure-store.

📂 src/services/

Purpose: Communication with external APIs and SDKs.

When to create: When you need to call Firebase, Supabase, or any backend.

Who uses it: Context, Hooks, Repositories.

Example files: auth.service.ts, user.service.ts

Common mistake: Putting UI logic in services. Services know nothing about React.

📂 src/repositories/

Purpose: Data access abstraction layer. Maps SDK responses to app types.

When to create: When services become complex and you need to swap data sources.

Who uses it: Services.

Example files: auth.repository.ts, user.repository.ts

Common mistake: Creating this folder when simple services suffice. Add it when needed.

📂 src/context/

Purpose: React Context definitions that provide global state.

When to create: When multiple screens need the same state.

Who uses it: Hooks (useAuth), Providers (RootProvider).

Example files: auth.context.tsx

Common mistake: Putting provider logic here. Providers go in providers/.

📂 src/hooks/

Purpose: Custom React hooks that encapsulate reusable logic.

When to create: When you extract logic from components.

Who uses it: Components, Screens.

Example files: useAuth.ts, useSession.ts

Common mistake: Hooks that do too much. One hook, one concern.

📂 src/providers/

Purpose: Compose and wrap all context providers for the app.

When to create: When you have multiple providers to nest.

Who uses it: app/_layout.tsx

Example files: RootProvider.tsx

Common mistake: Only using this folder if you have 1 provider. Skip if not needed.

📂 src/components/

Purpose: Reusable UI components that do not contain business logic.

When to create: When a UI element is used in more than one screen.

Who uses it: Screens.

Example files: LoginForm.tsx, SocialButtons.tsx

Common mistake: Components that call services directly. Components receive data via props.

📂 src/screens/ or app/ (Expo Router)

Purpose: Page-level components that users see and interact with.

When to create: When you need a new route/screen.

Who uses it: Navigation (Expo Router).

Example files: index.tsx (Login), register.tsx

Common mistake: Putting too much logic in screens. Screens orchestrate, they do not implement.

📂 src/navigation/

Purpose: Navigation configuration and route definitions.

When to create: When navigation logic grows complex (role-based routes).

Who uses it: app/_layout.tsx, protected route logic.

Example files: auth.navigation.ts

Common mistake: Over-abstracting navigation. Expo Router handles most of it.

Home / File Creation Order

File Creation Order

The exact order to create authentication files — and WHY each must come before the next.

Step 1: Create types/

Types define the contracts. Every other file depends on them. Create them first so TypeScript can guide your implementation.

💡 Why First?

Without types, you cannot define function signatures. You do not know what User looks like, what LoginCredentials expects, or what AuthState contains. Types are the blueprint.

Step 2: Create config/

Firebase/Supabase initialization. Services need this to exist. Create it before services.

Step 3: Create constants/

Token keys, error messages, default values. Services and storage use these.

Step 4: Create storage/

Secure persistence. Services need this to save tokens after login. Context needs this to restore sessions.

Step 5: Create services/

The core business logic layer. Depends on types, config, and storage. Everything above depends on services.

Step 6: Create repositories/ (optional)

Only when services need a data access abstraction. Not required for simple auth.

Step 7: Create context/

Global state. Depends on services (for login/logout actions) and storage (for session restore).

Step 8: Create hooks/

Clean API over context. Depends on context. Screens will use these.

Step 9: Create providers/

Compose context providers. Depends on context. Used by _layout.tsx.

Step 10: Create components/

Reusable UI. Uses hooks for data. Screens will compose these.

Step 11: Create screens/ (app/)

Final layer. Uses components + hooks. Navigation guards protect routes.

Step 12: Protect Navigation

Wire up _layout.tsx with auth state. This is the final piece that makes everything work together.

📝 Key Takeaways

  • Always start with types — they are the blueprint
  • Config and constants come before services
  • Storage comes before services (services need to persist)
  • Services come before context (context delegates to services)
  • Hooks come after context (hooks wrap context)
  • Screens come last (they compose everything)

🔑 Module: Login

The foundation of every authentication system.

Purpose

Login allows an existing user to authenticate with their credentials (email/password, phone, or social provider) and receive a session token.

Why It Exists

Without login, there is no way to verify identity. Every protected action, every personalized experience, every secure data access starts with login.

Required Files

  • app/index.tsx — Login screen
  • components/auth/LoginForm.tsx — Login form UI
  • hooks/useAuth.ts — Provides login() function
  • context/auth.context.tsx — Auth state management
  • services/auth.service.ts — Firebase/Supabase communication
  • storage/auth.storage.ts — Token persistence

Data Flow

1User Inputemail + password
2ValidationCheck format, required fields
3API CallsignInWithEmailAndPassword()
4TokenJWT returned from backend
5StorageSave to SecureStore
6ContextUpdate user state
7NavigationRedirect to Home

Code Example

LoginForm.tsx — Minimal TSX 
export function LoginForm() {
  const { login } = useAuth();
  const [email, setEmail] = useState('');

LoginForm.tsx — Production TSX 
export function LoginForm({ onSuccess }: { onSuccess?: () => void }) {
  const { login, loading, error } = useAuth();
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');
  const [validationError, setValidationError] = useState('');

  const handleSubmit = async () => {
    if (!email || !password) {
      setValidationError('All fields are required');
      return;
    }
    if (!email.includes('@')) {
      setValidationError('Invalid email format');
      return;
    }
    setValidationError('');
    await login(email, password);
    onSuccess?.();
  };

  return (
    <View>
      {validationError ? <Text style={{ color: 'red' }}>{validationError}</Text> : null}
      {error ? <Text style={{ color: 'red' }}>{error}</Text> : null}
      <TextInput value={email} onChangeText={setEmail}
        autoCapitalize="none" keyboardType="email-address" />
      <TextInput value={password} onChangeText={setPassword}
        secureTextEntry />
      <Button title={loading ? "Logging in..." : "Login"}
        onPress={handleSubmit} disabled={loading} />
    </View>
  );
}

Security Considerations

  • Never store passwords locally
  • Always use HTTPS (Firebase/Supabase handles this)
  • Implement rate limiting on the backend
  • Use secureTextEntry for password fields
  • Clear input fields after failed attempts

Common Mistakes

✅ Do
  • Show loading state during login
  • Display user-friendly error messages
  • Validate input before API call
  • Persist session in SecureStore
❌ Don't
  • Call Firebase SDK directly from UI
  • Show raw error messages to users
  • Store passwords in AsyncStorage
  • Skip input validation
🧠 Quick Quiz

Where should the Firebase signInWithEmailAndPassword call live?

Correct! The service layer (auth.service.ts) is the only place that should communicate with Firebase SDK. This keeps the UI clean and makes it easy to swap Firebase for another provider.

📝 Key Takeaways

  • Login is the foundation of authentication
  • The flow: Input → Validation → Service → SDK → Storage → Context → Navigation
  • Always show loading and error states
  • Never import Firebase SDK in UI components
  • Persist tokens in SecureStore, not AsyncStorage

🔄 Module: Session Restore

Keeping users logged in across app restarts.

Purpose

When the user opens the app again, restore their session from SecureStore so they do not have to login again.

Implementation

auth.context.tsx — Session Restore TypeScript 
useEffect(() => {
  const restoreSession = async () => {
    try {
      const token = await AuthStorage.getToken();
      if (token) {
        const user = await AuthService.getCurrentUser();
        if (user) setUser(user);
      }
    } catch {
      await AuthStorage.clear();
    } finally {
      setLoading(false);
    }
  };
  restoreSession();
}, []);

Home / Refresh Token

🔁 Module: Refresh Token

Keeping sessions alive without re-login.

Why Refresh Tokens?

Access tokens expire (e.g., after 1 hour). Refresh tokens are long-lived tokens used to get new access tokens without requiring the user to login again.

Token Lifecycle

Access Token (valid) → Token Near Expiry → Refresh Request → New Access Token

Implementation Strategy

💡 Proactive Refresh

Do NOT wait for the token to expire. Refresh it proactively when it is about to expire (e.g., 5 minutes before). This prevents failed API calls that confuse users.

hooks/useSession.ts — Auto Refresh TypeScript 
export function useSessionRefresh(token: string | null) {
  useEffect(() => {
    if (!token) return;

    const interval = setInterval(async () => {
      const newToken = await AuthService.refreshToken();
      if (newToken) {
        await AuthStorage.saveToken(newToken);
      }
    }, 50 * 60 * 1000); // Every 50 minutes (assuming 1h expiry)

    return () => clearInterval(interval);
  }, [token]);
}

🔒 Security Deep Dive

JWT, tokens, secure storage, OAuth, and production security.

JWT (JSON Web Token)

A JWT is a compact, URL-safe token that contains claims (data about the user). It has three parts:

JWT Structure Concept 
eyJhbGciOiJIUzI1NiJ9    // Header: algorithm used
.eyJ1aWQiOiIxMjM0NSJ9   // Payload: user data (uid, email, exp)
.sHd5ljKJgK9_VbR4mN2x   // Signature: ensures token integrity

Access Token vs Refresh Token

AspectAccess TokenRefresh Token
PurposeAuthorize API requestsGet new access tokens
LifetimeShort (15min - 1hr)Long (days - months)
Stored inMemory / SecureStoreSecureStore only
Sent toEvery API requestOnly when refreshing
If stolenLimited damage (expires soon)Full account access

Token Rotation

When a refresh token is used to get a new access token, the old refresh token should be invalidated and a new one issued. This limits the damage if a refresh token is compromised.

Secure Storage

✅ Secure (expo-secure-store)
  • Uses iOS Keychain / Android Keystore
  • Encrypted at rest
  • Biometric protection available
  • Never exposed to JavaScript bridge
❌ Insecure (AsyncStorage)
  • Plain text storage
  • Accessible to any code in the app
  • No encryption
  • Accessible on rooted devices

Security Checklist

  • ✅ Never store passwords in the app
  • ✅ Use HTTPS for all API calls
  • ✅ Store tokens in SecureStore, not AsyncStorage
  • ✅ Implement token expiration and rotation
  • ✅ Validate tokens on every API request
  • ✅ Use environment variables for API keys
  • ✅ Implement rate limiting on auth endpoints
  • ✅ Handle account lockout after failed attempts
  • ✅ Log authentication events for audit
  • ✅ Use deep link validation for OAuth callbacks
  • ✅ Clear all data on logout
  • ✅ Implement CSRF protection for web views

Common Attacks

Replay Attacks

An attacker captures a valid token and reuses it. Prevention: Short token lifetime, token rotation, request timestamps.

Brute Force Attacks

Attacker tries many password combinations. Prevention: Rate limiting, account lockout, CAPTCHA.

Token Theft

Attacker steals tokens from device storage. Prevention: SecureStore, device binding, short-lived tokens.

Session Hijacking

Attacker takes over an active session. Prevention: Token binding, IP monitoring, session invalidation on suspicious activity.

🎓 Interview Questions

Common authentication architecture interview questions and answers.

Q: Why do we need a service layer for authentication?

The service layer isolates external SDK dependencies from the UI. If we switch from Firebase to Supabase, only the service file changes — no UI code is affected. It also makes testing easier (we can mock the service).

Q: What is the difference between authentication and authorization?

Authentication answers "who are you?" (login). Authorization answers "what can you do?" (roles, permissions). Authentication comes first; authorization uses the authenticated identity.

Q: Why use SecureStore instead of AsyncStorage?

AsyncStorage stores data in plain text on the device. SecureStore uses the device's secure enclave (iOS Keychain, Android Keystore) which encrypts data at rest. Auth tokens are sensitive — if stolen, they can impersonate the user.

Q: How does token refresh work?

Access tokens have a short lifetime. When they expire, the client sends a refresh token to the server. The server validates the refresh token and issues a new access token. The refresh token may also be rotated (old one invalidated, new one issued).

Q: What is the purpose of the AuthProvider?

AuthProvider uses React Context to make authentication state (user, loading, error) available to the entire component tree without prop drilling. It wraps the app at the root level and provides login/logout functions.

Q: How do protected routes work?

Protected routes check the auth state (via useAuth hook). If the user is not authenticated, they redirect to the login screen. If loading, they show a spinner. If authenticated, they render the protected content.

Q: What is the repository pattern in authentication?

The repository abstracts data access. It maps between raw SDK responses and application types. If you switch from Firebase to a custom backend, only the repository changes — the service layer remains the same.

Q: How do you handle offline authentication?

Store the last known auth state and token in SecureStore. On app launch, check for a stored token. If the network is available, validate it with the server. If offline, use the cached state. Show a clear message when offline.

Q: Why separate context from providers?

Context defines the shape of the state (types, default values). Providers compose and supply that state. Separating them keeps files focused — context files define WHAT, provider files define HOW to provide it.

Q: How would you migrate from Firebase to Supabase?

Thanks to the service layer, you would: 1) Create a new Supabase config, 2) Rewrite auth.service.ts to use Supabase SDK, 3) Update types if needed, 4) Test all auth flows. No UI code changes needed.

Home / repositories/

📂 repositories/

Data access abstraction layer.

Repositories map between external SDK responses and your application types. They are an optional layer — use them when your services need to swap data sources or when SDK responses don't match your types.

💡 When to Use

For simple apps, services can handle data access directly. Add repositories when: you need to swap Firebase for another provider, your types differ significantly from SDK types, or you need to mock data access for testing.

Secure local data persistence.

Storage handles saving and retrieving sensitive data (tokens, user info) using expo-secure-store. This ensures data persists across app restarts securely.

🧭 Navigation

How navigation depends on auth state.

Auth-Aware Navigation

Navigation must show different screens based on auth state:

  • Not authenticated: Login, Register, Forgot Password
  • Authenticated: Home, Settings, Profile
  • Admin: Admin Panel (in addition to regular screens)

Implementation in Expo Router

app/_layout.tsx TSX 
function AuthStack() {
  const { user, loading } = useAuth();

  if (loading) return <LoadingScreen />;

  return (
    <Stack>
      {user ? (
        <>
          <Stack.Screen name="(home)" />
          <Stack.Screen name="(settings)" />
          {user.role === 'admin' && (
            <Stack.Screen name="(admin)" />
          )}
        </>
      ) : (
        <>
          <Stack.Screen name="index" />      {/* Login */}
          <Stack.Screen name="register" />
          <Stack.Screen name="forgot-password" />
        </>
      )}
    </Stack>
  );
}
